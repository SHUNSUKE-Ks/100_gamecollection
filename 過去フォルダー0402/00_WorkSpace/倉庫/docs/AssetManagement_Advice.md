# Library駆動開発における素材管理アドバイス (BGM・SE含む)

Library駆動開発（コンポーネントやアセットカタログを中心に据えた開発）において、素材管理は効率性と拡張性の要となります。
現在の `demo_assets_manifest.json` の構造（Screen単位）から、将来的なLibrary駆動（CategoryDB単位）への移行を見据えたアドバイスをまとめました。

## 1. 構造と管理方針

### 現状: Screen単位 (Wireframe的利用)
- **メリット**: 
    - 画面に必要なものを網羅的に洗い出せるため、**「必要な素材リスト（発注書）」**としてそのまま機能します。
    - 開発初期、画面を作る際に「どこに何があるか」が直感的で分かりやすいです。
- **Library駆動への視点**:
    - このJsonは、各画面の**「構成表」**として扱います。
    - 実装側では、このキー（ID）を使ってアセットを呼び出します。パスが変更されてもコード側の変更は不要になります。

### 将来: CategoryDB単位 (Library的利用)
- アセットが増えてきたら、`COMMON` のような分類を推し進め、カタログ化します。
- UI, Sound, BG, Sprite などの **「種類別 (Type)」** で管理する形に移行、または並行運用します。
- 例えば、`SoundManager` は全てのSE/BGMを知っている必要がありますが、画面ごとのローディング制御にはScreen単位の情報が役立ちます。

## 2. BGM・SE (Sound) の管理

サウンドも画像と同様に **「ID: Path」** で管理することが最も重要です。

- **命名規則**:
    - 接頭辞をつける: `SE_DECIDE`, `BGM_MAIN_THEME`
    - これにより、コード補完で `Assets.SOUND.` と打った時に探しやすくなります。
- **ダミー素材**:
    - 音は画像と違い「枠」が見えないため、無音やフリー素材で良いので「何かが鳴る」状態を早期に作ることが重要です。
    - `dummy_se.mp3` (短いビープ音など) を1つ用意し、未完のパスには全てこれを割り当てておくと、エラー落ちを防げます。

## 3. ダミー素材の運用 (SVG活用)

> 足りない画像は接頭詞（dummy）_ファイル名.svgで簡易的なものを作成

これは非常に優れたアプローチです。

- **SVGの利点**:
    - **色・サイズ・文字** をコードで簡単に生成・変更できます。
    - わざわざ画像ソフトを開かなくても、テキストエディタで「幅:100, 高さ:200, 色:グレー, テキスト:Hero」のようなSVGを作れます。
- **Libraryでの表示**:
    - Storybookや独自のカタログ画面（Library）で、このManifestを読み込み、一覧表示させます。
    - ダミー状態のものが一目でわかるため、「発注残」の管理表としても機能します。

## 4. プロパティの維持について

> Jsonですが、今はプロパティを増やさず｛識別ID：素材パス｝を維持してください。

この判断は正しいです。
- プロパティ（サイズ、オフセット、ループ設定など）をJsonに詰め込むと、管理が複雑化します。
- まずは「あるかないか」「どこにあるか」だけの **[ID: Path]** マッピングに徹することで、プログラム側は「読み込み」だけに集中できます。
- 詳細な設定が必要になった場合でも、別のJsonで `{"SE_TheCARD_FLIP": { "vol": 0.8 }}` のようにIDをキーにして拡張（マージ）する方が、データが汚れず柔軟です。
