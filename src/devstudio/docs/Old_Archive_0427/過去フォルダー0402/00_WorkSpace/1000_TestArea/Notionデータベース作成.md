LIB_Components（コンポーネント資産庫） UIやロジックの「実体」を管理します。 * **名前 (Title):** Prefix_Name（例: Base_MainButton） * **分類 (Select):** UI_Part, Component, Logic * **ステータス (Select):** Idea, In-Dev, Ready, Archived * **Props仕様 (Code):** TypeScriptの型定義（例: interface Props { ... }） * **外部依存 (Multi-select):** framer-motion, lucide-react 等（標準外のみ） * **参照JSON (Relation):** DEF_LayoutAssets と紐付け ### 2. DEF_LayoutAssets（レイアウト・アセット定義） Layout_SVG_Json の核となる、データ駆動設計の管理DBです。 * **Key ID (Title):** JSONのキー名（例: TUTORIAL_BG） * **シーン (Select):** Common, Tutorial, Battle, Home * **アセット種別 (Select):** SVG, Image, BGM, SE * **開発用ダミー (Files):** グリーンバック素材や軽量SVG * **本番パス (Text):** 最終的な画像パス（例: /assets/images/...） * **スケーリング設定 (Text):** 1920x1080 / aspect-ratio などの指示 ### 3. LIB_Hooks_Utils（共通ロジックDB） 状態管理や便利な関数を再利用するために保存します。 * **名前 (Title):** useFunctionName（例: useAudioPlayer） * **役割 (Select):** State(Zustand), Audio, Network, Calculation * **依存関係 (Relation):** 依存しているStoreや他のHook * **スニペット (Code):** コピペ可能な基本コード ### 4. DATA_GameMaster（ゲームドメインDB） skills.md や Agent.md の元データです。 * **ID (Title):** ユニークID（例: SKL_FireBall） * **カテゴリー (Select):** Skill, Agent, Item, Status * **パラメータ (JSON):** 攻撃力、コスト、継続時間などの数値データ * **演出リレーション (Relation):** 紐付ける演出用 LIB_Components ### 5. DOC_ProjectStandards（プロジェクト共有ルール） 技術スタック（Package.json）やマニュアルを管理します。 * **ドキュメント名 (Title):** Technical Stack, Coding Standards, v1.4 Folder Structure * **更新日 (Date):** 最終更新日 * **対象 (Multi-select):** Programmer, Designer, Planner * **内容概要 (Text):** 「状態管理はZustandで行う」「画像はグリーンバックで用意する」等の核心的ルール --- ## 🛠 プログラマーへの指示出し：コンポーネント記述テンプレート案 DB内の「ページ」の中に以下の項目を用意しておくと、プログラマーが即座に実装に入れます。 > ### [Component Name] 実装依頼票 > > > 1. **目的:** このコンポーネントで何を達成するか > 2. **参照JSONキー:** DEF_LayoutAssets のどのキーを使うか > 3. **Props定義:** プログラマーが受け取るデータの型 > 4. **アクション:** ボタン押下時などのコールバック関数 > 5. **表示条件:** Zustand のどの値を監視して表示/非表示を切り替えるか > > このDB構成で構築を開始しましょうか？それとも、まずは特定のDB（例えば DEF_LayoutAssets）の詳細な記入例から固めていきますか？
