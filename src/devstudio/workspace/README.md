# DevStudio WorkSpace — 設計README

> **「WorkSpaceはタスクが決める」**
> 作業画面はあらかじめ決まっていない。タスクが発生したとき、そのタスクに最適な作業形態を選択する。

---

## 1. WorkSpaceとは何か

WorkSpaceは「状態を変換する場所」です。

```
Dashboard（俯瞰・確認）
  └─ WorkSpace（実際に手を動かす場所）
        └─ タスクごとに作業形態が変わる
```

Dashboardが「何をすべきか」を示すのに対し、WorkSpaceは「実際にそれをする場所」です。
シナリオを書く・ComfyUI発注をする・設計を決める、といった作業は全てここで行います。

---

## 2. Human Decision Task（人間への決定依頼）

### 概念

AIが処理を進める際に人間の判断が必要な場合、AIは処理を止めず、**Taskを作成してWorkSpaceに投げます。**

```
AI処理中...
  ↓ 「ここは人間が決める必要がある」
  ↓
[Task作成] type: decision
  title: 「〇〇の方針を決めてください」
  → WorkSpaceに Decision フォームとして表示
  ↓
人間がWorkSpaceで記入・選択
  ↓
state更新 → AIが次の処理へ
```

### メリット

- AIは「決定待ち」で止まらない（他のタスクを進められる）
- 人間は「何を決めればよいか」が明確になる
- 決定内容がstateに記録されるので後から参照できる

---

## 3. Decision Task フォーマット

### Task の type フィールド（拡張予定）

| type | 内容 |
|------|------|
| `implement` | 通常の実装タスク |
| `decision` | 人間の判断が必要な決定依頼 ← 今回追加 |
| `review` | レビュー・確認依頼 |
| `research` | 調査・情報収集 |

### Decision フォームの構造（MD ベース v1）

```markdown
# [決定依頼] タスクタイトル

## 背景
なぜこの決定が必要か、コンテキストを記述。

## 選択肢
- [ ] A: 〇〇する（メリット: xxx / デメリット: xxx）
- [ ] B: △△する（メリット: xxx / デメリット: xxx）
- [ ] C: その他 → 記入欄↓

## 記入欄
（自由記述）


## 制約・前提
- 〇〇という制約がある
- △△は変更不可

## この決定が影響するもの
- state.xxx
- T-005 の実装方針
```

### WorkSpace での表示イメージ

```
┌─────────────────────────────────────────┐
│ 🔷 [決定依頼] DevLog v2 の timestamp 型  │
│                                          │
│ 背景 ────────────────────────────────    │
│ 現在 string (ISO) か number (epoch) か   │
│ 決まっていない。マニュアルは number。     │
│                                          │
│ 選択肢 ──────────────────────────────    │
│ ☐ A: number (epoch ms) — 推奨           │
│ ☐ B: string (ISO 8601) — 現行互換       │
│ ☐ C: 両方持つ                            │
│                                          │
│ 記入欄 ──────────────────────────────    │
│ ┌─────────────────────────────────────┐ │
│ │                                     │ │
│ └─────────────────────────────────────┘ │
│                                          │
│              [決定を確定する →]           │
└─────────────────────────────────────────┘
```

## 4. WorkSpace の動的な作業形態

タスクの種類によってWorkSpaceに表示される「作業UI」が変わります。
これはあらかじめ全種類を実装するのではなく、**タスクが発生するたびに設計します。**

### 現在想定している作業形態

| 作業タイプ | WorkSpace UI | 状態変換の内容 |
|-----------|-------------|--------------|
| `decision` | Decision フォーム（MD形式） | 未決定 → 決定済み |
| `scenario` | シナリオエディタ（将来） | プロット → SCENARIO.json |
| `comfyui` | ComfyUI発注フォーム | プロンプト → 画像生成キュー |
| `review` | Diff ビューア（将来） | 未レビュー → 承認/差し戻し |

### 追加のルール

新しい作業形態が必要になったら：
1. タスクが発生した段階でAIとユーザーが相談
2. その作業に必要な最小限のUIを設計
3. DevStudioの既存思想（状態変換モデル）に合うか確認してから実装

## 5. Today's Tasks（本日の人間タスク）

DashboardのPlannerビュー1段目に表示される「本日の人間タスク」は、WorkSpaceへの入口です。

```
Today's Tasks
┌────────────────────────────────────────┐
│ ▶ [decision] DevLog timestamp 型を決める│  ← クリック
│ ○ [implement] T-004 型定義改訂開始     │     → WorkSpaceの
│ ○ [review] スキーマv2 確認             │        該当タスク作業画面へ
└────────────────────────────────────────┘
```

---

## 6. 今後の実装順序

```
Phase 1（現在）: Decision フォーム（MD形式）
  → シンプルな記入欄 + チェックボックス
  → 決定内容を Task.output に保存

Phase 2: 動的フォームレンダラー
  → Task の workspaceConfig を読んでUIを生成

Phase 3: 各作業タイプの専用UI
  → シナリオエディタ、Diff ビューアなど随時追加
```

---

*WorkSpace は「決まった場所」ではなく「決める場所」です。*
*タスクが持つ意図（intent）が、そこに表れる作業形態を決定します。*
